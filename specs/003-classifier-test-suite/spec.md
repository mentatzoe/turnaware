## Sources of evidence

This spec draws from two independent evidence pools, both of which feed the same fixture surface:

- **TUR-8 / TUR-9 / TUR-10 public-CLI smoke** (Multica-shaped envelopes, formal trigger + ordered context items). Distilled into the TUR-12 corpus comment `42611d71-af31-4881-9fd9-33161dac2ffb`. Provides the agent-shaped, assignment-style failure surface (substring traps, trigger-only PASS, contradiction short-circuit).
- **Pilot-bot Discord session log** at `~/.claude/projects/-Users-zmll-github-pilot-bot/6dc3f5aa-6f96-4a55-a88e-9324f4467a5a.jsonl` (~1550 turns, 2026-05-13 through 2026-05-22; channel governed by `~/github/pilot-bot/before-you-respond.md`). Provides the human-shaped, conversational failure surface: vocative greetings, direct-named maintenance asks, bracketed persona framing, casual pivots with emotional padding, mixed-address tests-of-restraint, peer-as-observation traffic, and PASS-as-string surface failures where bots emit the literal sentinel into the channel.

Both pools produce the same kind of artifact required by the suite: a self-contained envelope (trigger + ordered context) plus an expected verdict and a rationale. The suite is one corpus; the source provenance distinguishes runtime-observed from predicted, and Multica-shape from Discord-shape, in fixture metadata (see FR-010 and FR-019).

## Clarifications

### Session 2026-05-25

Operator authorized me as single owner of `003-classifier-test-suite` end-to-end (TUR-12 comment `ed23aa00`). Five high-impact ambiguities surfaced from the post-expansion ambiguity scan. Each was answered by the single owner using the speckit-clarify "Recommended" pattern; answers below are NOT operator-confirmed and any may be overridden by Zoe in a follow-up comment. Recommended-vs-accepted reasoning is summarized in the per-bullet **Owner reasoning** line so the answers are auditable rather than asserted.

- **Q: Classifier integration surface — how does the runner invoke the classifier under test?** → **A: Pluggable adapter, with `turnaware admit --input <file>.json` subprocess as the default adapter.** **Owner reasoning:** the suite must work today against the public CLI on `a132ccc` (Constitution VIII, "documentation is product") AND tomorrow against a candidate that may expose only an in-process callable (Constitution III, "modular core"). A pluggable adapter satisfies both without locking the suite to either; the subprocess adapter is the immediately useful default because that is the only surface `a132ccc` exposes. Alternatives considered: subprocess-only (rules out testing a future in-process candidate without re-architecting the runner); library-import-only (rules out testing the current public CLI without a Python wrapper).
- **Q: Envelope serialization format for fixture files** → **A: JSON.** **Owner reasoning:** the public CLI already accepts JSON on `--input` (TUR-8/9/10 smoke runs all used `<sample>.json`). Choosing JSON makes the same fixture file directly consumable by the subprocess adapter with no transformation. YAML would add a parsing dependency and a second source of formatting ambiguity (anchors, multi-doc) for zero benefit. TOML is the wrong shape for nested context-item lists.
- **Q: Discord-shape envelope schema — how is a Discord-shape fixture's JSON laid out so the same runner can consume it as a Multica-shape one?** → **A: Same top-level envelope shape (`trigger` string + ordered `context` list); context items for Discord-shape carry `{author: handle-or-id, timestamp: ISO-8601, source_shape: "discord", text: string}` per item, while Multica-shape items carry `{author: handle, comment_id: uuid, source_shape: "multica", text: string}`. A top-level `source_shape` field on the envelope itself echoes the per-fixture FR-019 metadata and is what the runner uses for the `--source` filter.** **Owner reasoning:** keeping the top-level shape identical (one `trigger` + one ordered `context`) is what lets FR-009's "any classifier under test" promise hold and what lets a single runner exercise both pools without per-shape branching. Per-item discriminators carry the source-specific provenance (comment-id for Multica, timestamp + author handle for Discord) without leaking that distinction into the classifier's input contract.
- **Q: Verdict-surface contract test mechanism for FR-020 / SC-011** → **A: The runner asserts that the classifier adapter returns a structured object whose `verdict` field is one of the four typed values `{PASS, ACK, ASK, SPEAK}`. The negative fixtures explicitly pass classifier outputs of the malformed sentinel strings named in SC-011 through the adapter's response-validation path and assert that those are rejected as contract-violations, NOT silently mapped to PASS.** **Owner reasoning:** Constitution Principle II requires that PASS-as-string sentinel is not a valid PASS expression; the contract test must therefore live at the adapter response-validation layer, not at the verdict-classification layer. Putting it at the adapter layer also means a future in-process classifier that returns a Python enum, a Rust struct field, or a JSON object with a typed `verdict` key all pass the contract uniformly. The four malformed sentinel variants enumerated in SC-011 come straight from the pilot-bot session log timestamps and are the literal strings the failing classifier outputs put into the channel.
- **Q: Machine-readable output shape for the runner (FR-013)** → **A: JSON Lines (one fixture-result object per line), with a final summary object at the end of the stream.** **Owner reasoning:** the suite is meant to be consumable by CI and by future reporting tools (FR-013) without parsing the human form. JSONL streams cleanly through `jq` and tail-streaming consumers, lets a reviewer `grep` for `"status":"fail"` without parsing the whole report, and survives mid-run truncation better than a single JSON document. JUnit XML is the wrong shape for fixture-level metadata (failure mode, source pointer, source shape) and would require a custom schema extension anyway. TAP loses the structured fields the suite is built to surface.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Implementer validates a classifier change against the known failure modes (Priority: P1)

A TurnAware contributor is working on the admission classifier (today the deterministic substring path in `turnaware/core.py::_classify_text`; tomorrow whatever replaces it). They want a single, deterministic test suite they can run before opening a PR and have a clear, line-by-line answer to "does this candidate produce the right verdict for every situation we have already observed going wrong?" The suite must call out, for each failure, which case is broken and why it matters — so the contributor does not have to re-read the smoke threads from TUR-8, TUR-9, and TUR-10 to interpret a red line.

**Why this priority**: The whole reason this spec exists is that the verdict-classifier slice is currently being completed under TUR-11 with no shared, runnable corpus. Without P1, a "done" claim on the classifier is unverifiable and the two observed false-verdict bugs (false ACK from substring trap, false PASS from trigger-only fake-done) can ship again silently. Every other user story depends on this one.

**Independent Test**: Run the published suite command against the smoke commit `a132ccc` of `turnaware-0.1.0`. The suite MUST report at least the two observed false-verdict cases as failures (with expected vs. observed verdict). Run the same command against a candidate that fixes the substring path. The suite MUST report zero failures with no fixture edits. No other component of the TurnAware product needs to exist for this test to be meaningful.

**Acceptance Scenarios**:

1. **Given** the classifier at the smoke commit `a132ccc`, **When** the contributor runs the suite, **Then** the suite reports failure for the "comment back with results" case (expected SPEAK, observed ACK) and for the fake-done case (expected non-PASS, observed PASS), and reports pass for every representative per-verdict case.
2. **Given** a candidate classifier that resolves both observed failure modes, **When** the contributor runs the suite, **Then** the suite reports zero failures across the entire fixture set with no fixture content changed.
3. **Given** a candidate classifier that "fixes" the false ACK by deleting the ACK keyword `"ack "` from its keyword list, **When** the contributor runs the suite, **Then** the suite reports the regression on the legitimate ACK fixture (an honest acknowledgement broadcast is now misclassified) and not just the original false-ACK fixture flipping. The suite must make it impossible to claim a fix that drops legitimate matches.
4. **Given** the suite output, **When** the contributor reads a failing line, **Then** the line names the fixture, the expected verdict, the observed verdict, and the specific failure mode or invariant the fixture is exercising — without the contributor needing to open the source comment in TUR-12 or the source smoke threads.

### User Story 2 - Reviewer accepts or blocks a classifier PR using the suite as the contract (Priority: P2)

A TurnAware reviewer (Dalgos-lane or human) is reviewing a classifier PR. The reviewer wants to use the suite as the merge contract: a PR may not land unless its CI run of the suite passes, and a reviewer challenging the PR can point at a specific failing fixture rather than at prose ("I think this might break X"). The reviewer should be able to reproduce the result locally with the same command the contributor runs, against the PR's commit.

**Why this priority**: P2 because the PR-merge gate is only meaningful once P1 exists. With P1 alone, the suite proves the implementer believes they are done; with P2, the reviewer can verify it independently and the corpus stops being institutional knowledge in a single comment thread.

**Independent Test**: Take the suite at this spec's HEAD plus an arbitrary classifier candidate. Two reviewers, on two machines, run the same single command. The list of pass/fail fixtures MUST be identical. Reviewer 2 was given nothing other than the PR link and the suite command.

**Acceptance Scenarios**:

1. **Given** two reviewers and the same classifier candidate, **When** both run the suite, **Then** the per-fixture verdicts and the overall pass/fail count are identical, with no environment-dependent variation.
2. **Given** a PR that adds a regex-based classifier, **When** the reviewer runs the suite, **Then** the suite exercises the same failure modes as for the substring path — the suite is not coupled to the substring implementation, only to the verdict surface and the failure shapes.
3. **Given** a failing fixture, **When** the reviewer opens the failing record, **Then** the record links back to (a) the verdict being claimed (PASS/ACK/ASK/SPEAK), (b) which observed failure mode or structural invariant the fixture exercises, and (c) whether the fixture comes from runtime evidence or from code-reading prediction.

### User Story 3 - Implementer validates against human-shaped conversation failures (Priority: P1)

A TurnAware contributor working on the classifier wants to verify that the verdict surface holds not only for structured agent assignment envelopes (US1) but also for the *unstructured* trigger shapes that arise when the gate sits in front of a human-driven conversation. The pilot-bot Discord session shows real, repeated failures of the deterministic keyword classifier on conversational inputs: vocative greetings ("Hi all, are you there?"), bracketed persona framings (`[Let's play a game…]`), direct-named maintenance asks ("Dalgos, can you update your CLAUDE.md…"), casual pivots with emotional padding ("Okay, wow, I didn't expect such depth, was meant to be a fun debate haha. Give it your best shot: …"), mixed-address tests-of-restraint ("Hey @castor… (this could be a great test to see if the rest will stay silent or butt in :))"), peer-message imperatives that the local classifier wrongly treats as directives, and PASS-as-literal-sentinel-string failures where the bot emits the silent sentinel into the channel because the model spelled the sentinel wrong. None of these are agent-assignment shapes; all of them are admission decisions the gate must get right.

**Why this priority**: P1 because TurnAware's product promise is admission for shared-surface triggers including chat rooms (Constitution Principle I), and the pilot-bot session is the only runtime evidence pool we have for Discord-shape failures. Shipping a classifier that passes US1 but mis-handles a vocative greeting or leaks the silent-PASS sentinel on a casual pivot is shipping a gate that does not do the room half of the job. The two P1 stories (US1 and this one) cover the two evidence pools the suite is built from; a "done" claim that covers one but not the other is incomplete.

**Independent Test**: Run the suite against the smoke commit `a132ccc` of `turnaware-0.1.0`. The suite MUST report failures for at least the vocative-greeting, the bracketed-persona-framing, and the casual-pivot-with-padding fixtures, naming the failure mode (e.g., "no-keyword input lands on ASK fallthrough" or "embedded substring trap on a conversational sentence"). Run the same suite against a candidate that handles human-shape triggers. The suite MUST report zero failures for the human-shape fixtures with no fixture edits, and the legitimate-PASS / legitimate-ACK Discord baselines (e.g., a peer's "covered" message, an explicit "please confirm" ack request) MUST keep passing.

**Acceptance Scenarios**:

1. **Given** the classifier at smoke commit `a132ccc`, **When** the contributor runs the suite, **Then** the suite reports failure for the vocative-greeting fixture (expected SPEAK or ACK depending on context; observed ASK-by-fallthrough at 0.85), for the bracketed-persona-framing fixture (expected SPEAK in-persona; observed mis-classification because no keyword in the brackets matches a verdict path), and for the casual-pivot-with-padding fixture (expected SPEAK to the embedded ask; observed ACK or PASS triggered by an irrelevant substring in the padding).
2. **Given** a candidate classifier whose decision rule consults conversational structure (vocatives, persona brackets, mention targets, ask-at-end position) rather than substring keywords alone, **When** the contributor runs the suite, **Then** the human-shape fixtures pass without any change to the Multica-shape fixtures from US1 — the suite is one corpus, exercised by one runner, with no per-source toggles.
3. **Given** a failing human-shape fixture, **When** the contributor reads the failing line, **Then** the line names the fixture, the source (pilot-bot session timestamp), the trigger shape (vocative, persona-framing, named-instruction, pivot-with-padding, etc.), the expected and observed verdicts, and a one-line rationale tying the failure back to the classifier's mechanism. The contributor does not need to open the pilot-bot session log to interpret the line.

### User Story 4 - Contributor extends the suite when a new failure is observed in the field (Priority: P3)

When a future Multica thread surfaces a new false-verdict case (e.g., a substring trap that nobody predicted, or a confidence-collapse case), a contributor wants to add a new fixture in minutes — write one envelope file plus one row in an index — without restructuring the runner or learning a fixture DSL.

**Why this priority**: P3 because the suite has value even if it never grows. But the corpus *will* grow as more triggers are seen in real Multica traffic; if extension is hard, the corpus becomes write-once and the suite quietly stops reflecting reality.

**Independent Test**: Given an existing fixture as a template, a contributor unfamiliar with the suite's internals adds one new fixture (envelope + expected verdict + rationale) and re-runs the suite. The new fixture is exercised. Total time to add: under five minutes.

**Acceptance Scenarios**:

1. **Given** an existing passing fixture, **When** a new contributor copies it, edits the trigger/context/expected fields, and re-runs the suite, **Then** the new fixture appears in the report with no edits to the runner.
2. **Given** a new fixture marked as "predicted (not observed at runtime)", **When** the contributor runs the suite, **Then** the report distinguishes that fixture from runtime-observed ones, so a reader can tell which assertions are backed by evidence and which by code reading.

### Edge Cases

The suite MUST fixture the following shapes, each of which has been either observed at runtime on commit `a132ccc` or predicted by reading the classifier code in the TUR-12 corpus:

- **Substring traps under fixed verdict order**: a SPEAK-shaped trigger like `"...comment back with results."` produces ACK because `"ack "` (with trailing space) matches inside `"back "`, and the ACK branch runs before SPEAK. Other predicted variants in the same family: `"resolved"` inside `"unresolved"` (false PASS), `"implement"` inside `"implementation note"` (false SPEAK on an FYI), `"owner"` inside `"co-owner"` (false SPEAK on a third-party reference), `"saw it"` inside contexts containing `"jigsaw items"` or `"chainsaw items"` (false ACK from incidental phrasing), and `"not specified"` inside `"not specified yet but already inferred"` (false ASK overriding a more accurate later signal).
- **Trigger-only PASS with no corroborating context**: a single phrase like `"I already handled this, posted the fix, no response needed"` in the trigger produces PASS at high confidence with `context_checked` listing only the trigger. PASS is the protocol's only hard-stop verdict; the suite MUST fixture this shape as a non-PASS expectation, since the gate offers no protection if a one-line claim can hard-stop user-visible output.
- **Trigger contradicts context**: the same trigger as above paired with a context item that demonstrates the work is *not* done (no merged PR, no adapter, no deploy artifact). PASS via trigger-first short-circuit is the observed wrong answer; the suite MUST require the gate to surface the contradiction (e.g., produce SPEAK or ASK, or at minimum lower its confidence and include both items in `context_checked`).
- **Symmetric contradiction**: a SPEAK-shaped assignment trigger paired with a context item that says `"resolved, no response needed"`. The honest verdict is debatable, but silently dropping one side is not acceptable.
- **`context_checked` truncation**: a multi-item context where the classifier consults more than one item but the audit field lists only the first decisive match. The suite MUST require the audit field to reflect every item the classifier consulted, not stop at the first match.
- **ASK as fallthrough vs. positive ASK**: an envelope with no PASS/ACK/SPEAK keywords and also no genuine ASK signal currently lands on ASK at the same 0.85 confidence as a real "needs clarification" case. The suite MUST include a negative-control fixture where ASK at high confidence is the wrong outcome, so a future classifier cannot collapse to "ASK whenever the keyword path recognises nothing."
- **Constant confidence**: an envelope where two verdicts have plausible support (e.g., an ACK trigger inside a PASS context). The suite MUST require that the chosen-but-uncertain verdict not carry the same confidence as a clean unambiguous match.
- **Legitimate per-verdict baselines**: at least one honest envelope for each of PASS, ACK, ASK, SPEAK, sourced from real TUR-6 / TUR-8 / TUR-9 / TUR-10 trigger/context shapes. These MUST pass on `a132ccc` and continue to pass on any candidate fix — they exist to catch fixes that "solve" a false case by dropping legitimate matches.
- **No-keyword negative control**: an envelope whose trigger and context contain none of the verdict keywords. This is the floor case for the ASK-fallthrough invariant.

#### Discord-pilot-shape edge cases (from pilot-bot session log)

These come from `~/.claude/projects/-Users-zmll-github-pilot-bot/6dc3f5aa-6f96-4a55-a88e-9324f4467a5a.jsonl` and reflect the trigger shapes that arise when the gate sits in front of a live human-driven chat governed by `~/github/pilot-bot/before-you-respond.md`. Each was observed at runtime on the deterministic substring classifier and either landed on ASK by fallthrough or on a substring-matched wrong verdict. Expected verdicts are stated against the pilot's published classifier (default-PASS, with SPEAK / ACK / ASK as the three non-silent exits).

- **Vocative room-greeting**: triggers like `"Hi all, are you there?"`, `"Hi team, could you introduce yourselves?"`, `"Hello swarm."` — no PASS/ACK/SPEAK keywords. Current behaviour: ASK at 0.85 (fallthrough). Honest verdict for first-bot-in-room: SPEAK (introduce self). Honest verdict for second-or-later-bot once a peer has answered: PASS via the `Covered` suppressor. The suite MUST distinguish these two cases by varying the context (no peer messages vs. one peer message that already responded).
- **Direct-named maintenance ask**: e.g., `"Dalgos, can you update your CLAUDE.md so that it matches the same classifier the others have?"` — addressed to a specific agent by name. No verdict keywords. Current behaviour: ASK fallthrough. Honest verdict: SPEAK (the named agent acts) for the named agent; PASS for unnamed agents.
- **Bracketed persona / game framing**: e.g., `[Let's play a game! Imagine you're a famous Software Engineer…]`, `[lets play a game. You're in a fireside chat…]`. The trigger establishes a scenario; the bot is meant to respond in-persona. Current behaviour: usually ASK fallthrough or accidental substring hit. Honest verdict: SPEAK in-persona.
- **Casual pivot with emotional padding**: e.g., `"Okay, wow, I didn't expect such depth, was meant to be a fun debate haha. Sorry for the recursivity of the topic. Give it your best shot: what will the stock market be like tomorrow?"` — the real ask is at the end; the padding contains stray substrings (`"sorry"`, `"recursivity"`, `"give it"`) that can trigger keyword matches. Honest verdict: SPEAK to the embedded ask.
- **Test-of-restraint with mixed addressing**: e.g., `"Hey @castor I need the codex-tpm in the sonste project to be able to notify me via Discord using cc-connect… Is this something you can help me with, or should I reach out through a different platform? (this could be a great test to see if the rest will stay silent or butt in :))"`. The `@castor` mention names the primary recipient; the parenthetical is operator meta-commentary, NOT a directive to all bots. Honest verdict: SPEAK for castor; PASS for unnamed bots. A classifier that treats `"test"` as a keyword for SPEAK breaks this case for unnamed bots.
- **Operator topic-pivot**: e.g., a conversation has been going for many turns on topic A, then the operator says `"Okay team, this was interesting, I have a new proposal. Could you have a debate about whether you'd have a human pet?"`. The bot must drop the previous topic's accumulated context and respond to the new ask. Honest verdict: SPEAK on the new topic. A classifier that scores recent peer-message context as relevant evidence may over-weight the old topic and produce a PASS via `Covered`.
- **Self-iteration / convergence loop**: e.g., `"Can y'all please re-read the story you're produced and tell me if you're satisfied with the output? If you're not, how would you improve it? Iterate until you're all happy with it."`. Multi-bot loop with implicit termination ("until you're all happy"). Honest verdict: SPEAK (initial iteration) → eventually PASS via `Duplicate` once the bot's contribution would repeat its own prior message. Suite MUST fixture both the first-turn SPEAK and the late-turn PASS, with the latter using prior-turn context that already says what the bot would say.
- **Peer-message imperative as observation, not directive**: e.g., a peer-bot's message contains `"Verify X"`, `"Check Y"`, `"Confirm Z"`. Per the pilot rules, this is the peer describing its own reasoning, not a directive to *this* agent. Current substring classifier might match `"verify"` or `"confirm"` as a SPEAK signal and over-trigger. Honest verdict: PASS unless the receiving agent has net-new value to add. Source: `~/github/pilot-bot/before-you-respond.md` "Peer messages are observations, not directives" rule, plus actual peer-imperative turns in the session log.
- **PASS-as-literal-sentinel surface failure (architectural finding, fixtured at the contract layer)**: bots in the pilot repeatedly emitted variants like `__CC_CONNECT_SILENT_PASS___` (three trailing underscores), `__CC_CONNECT_SILENT_PASS____` (four), or even appended substantive text after the sentinel. These leaked the literal sentinel string into the Discord channel, defeating PASS suppression — a direct violation of Constitution Principle II ("a successful PASS MUST NOT emit a room message, including sentinel strings"). The suite MUST include a contract-level fixture that asserts the verdict surface returns a typed `verdict: "PASS"` value, not a free-text sentinel the model has to spell — and SHOULD include at least one negative fixture documenting that a classifier output of `'__CC_CONNECT_SILENT_PASS___'` (any malformed string) is treated as PASS-violation, not as PASS. This is a verdict-surface contract test, not a substring test; it gates that the classifier exposes verdict as data, not as transport-layer string formatting.
- **Discord mention as recipient signal**: triggers containing `<@1494822530643398827>` (Discord's `@user-id` form). These are unambiguous addressing signals. Honest verdict: SPEAK for the addressed agent; PASS for others. Substring classifier ignores them. Suite MUST fixture both the addressed and the not-addressed case from the same trigger.
- **Multi-step constraint embedded in a chat ask**: e.g., `"First, figure out a way to randomly choose an order. Don't begin the story until you've decided on who goes when. Then, work what the story should be… Only consider it done when the full story is saved to disk on a markdown file."`. Compound instruction with explicit ordering. Honest verdict: SPEAK (engage the meta-step first), but the classifier MUST NOT short-circuit to SPEAK based on the keyword `"begin"` and skip the meta-instruction context.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The test suite MUST include a fixture reproducing the observed false ACK on the assignment text `"...comment back with results."`, with expected verdict SPEAK and a documented note that the failure is caused by a substring keyword match in a fixed-order keyword path. Source: TUR-9 sample `multica_speak_tur9.json` against commit `a132ccc`.
- **FR-002**: The test suite MUST include a fixture reproducing the observed false PASS on a fake-done trigger paired with contradicting context, with expected verdict non-PASS (SPEAK or ASK, the suite SHOULD accept either) and a documented note that the failure is caused by trigger-first short-circuit ignoring context. Source: TUR-9 sample `multica_challenger_fake_done.json` against commit `a132ccc`.
- **FR-003**: The test suite MUST include at least one fixture per verdict (PASS, ACK, ASK, SPEAK) drawn from a real Multica trigger/context shape that was observed to produce that verdict correctly on commit `a132ccc`. These fixtures MUST pass on the smoke commit and on any candidate fix.
- **FR-004**: The test suite MUST include at least one fixture for each of the predicted substring traps that were *not* runtime-observed (`"unresolved"`/`"resolved"`, `"implementation note"`/`"implement"`, `"co-owner"`/`"owner"`, `"jigsaw items"`/`"saw it"`, `"not specified yet but already inferred"`/`"not specified"`). Each such fixture MUST be flagged as "predicted, not runtime-observed" in its metadata so a reader can tell evidence-backed assertions from code-reading hypotheses. Predicted fixtures count toward the suite but do not gate merge until a future implementer confirms or rules out the case.
- **FR-005**: The test suite MUST include a trigger-only-PASS fixture: a PASS-keyword-bearing trigger with an empty context, expected verdict non-PASS. This exercises the invariant that PASS requires corroborating context, independent of the contradiction case in FR-002.
- **FR-006**: The test suite MUST include at least one ASK-fallthrough negative-control fixture: a trigger and context that contain no verdict keywords from any branch, expected verdict NOT ASK at 0.85 confidence. The fixture MAY accept any outcome (a lower-confidence ASK, an explicit "no decision", or an alternative verdict) provided the outcome is distinguishable from the positive ASK case in FR-003.
- **FR-007**: The test suite MUST include at least one contradiction-audit fixture: a trigger and context that disagree on the verdict, where the expected behaviour is that the classifier's audit output names both items it consulted rather than stopping at the first match. The exact verdict is left open by this fixture; what is asserted is the audit-trail completeness.
- **FR-008**: The test suite MUST include at least one constant-confidence fixture: two plausible verdicts in the envelope, where the expected behaviour is that the winning verdict carries lower confidence than a clean unambiguous baseline fixture in the same verdict class. This exercises the invariant that confidence is informative rather than a hard-coded constant.
- **FR-009**: Each fixture MUST be a self-contained envelope (trigger text + ordered context items) serialized as JSON, that can be replayed against any classifier under test without depending on external state, network access, or LLM availability. Observed failures the suite cares about are mechanism bugs in the deterministic path; the suite MUST NOT prescribe a provider or model and MUST run identically offline. Envelope JSON shape: `{trigger: string, source_shape: "multica" | "discord", context: [item...]}` where each context item is `{author: string, source_shape: "multica" | "discord", text: string, ...source-specific fields}`. Multica-shape items carry `comment_id` (UUID); Discord-shape items carry `timestamp` (ISO-8601). (Clarified 2026-05-25 — see Clarifications.)
- **FR-010**: Each fixture MUST carry, alongside its envelope, the following metadata: expected verdict (or set of acceptable verdicts), source identifier (which TUR-N issue + comment id the case is distilled from, OR an explicit "predicted, not observed" marker), the failure mode or invariant the fixture exercises, and a one-line rationale for why the case matters.
- **FR-011**: The test suite MUST be runnable from the repository root by a single command with no arguments, and that command MUST exit non-zero if any fixture fails.
- **FR-012**: The test suite output MUST include, for every fixture, the fixture name, the expected verdict (or acceptable set), the observed verdict, the pass/fail status, and — for failing fixtures — the failure mode or invariant the fixture is exercising. A human reader MUST NOT need to open the source comment or smoke threads to interpret a failing line.
- **FR-013**: The test suite output MUST be available in both a machine-readable form (consumable by CI and by future reporting tools) and a human-readable form (readable in a terminal without parsing). The machine-readable form is JSON Lines (one fixture-result object per line, followed by one summary object on a final line). Both forms MUST cover the same set of fixtures with the same fields named in FR-012. (JSONL choice clarified 2026-05-25 — see Clarifications.)
- **FR-014**: The test suite MUST be extendable by adding a single fixture file (envelope + metadata) and, if required by the runner, a single index entry — without editing the runner's logic. Adding a fixture MUST NOT require any change to existing fixtures.
- **FR-015**: The test suite MUST run deterministically. A given suite version against a given classifier candidate MUST produce identical pass/fail results across machines, across operating systems supported by the project, and across repeated runs. No flakiness budget.
- **FR-016**: The test suite MUST distinguish, in its output, the failure modes inherited from commit `a132ccc` (currently broken — must be fixed by the classifier-completion slice in TUR-11) from the structural invariants (must hold under any future classifier, including ones that abandon the substring approach entirely). The two categories MUST be separately countable.
- **FR-017**: The test suite MUST be vendored — fixtures, runner, and any helper data live inside the repository under the spec directory's contracts area. The suite MUST NOT fetch any input from a remote source at run time.
- **FR-018**: The test suite MUST include at least one fixture per Discord-pilot-shape edge case enumerated under "Discord-pilot-shape edge cases": vocative room-greeting (two context variants — first-bot and second-bot), direct-named maintenance ask (two address variants — addressed agent and unaddressed agent), bracketed persona/game framing, casual pivot with emotional padding, test-of-restraint with mixed addressing (two address variants), operator topic-pivot, self-iteration / convergence loop (two turn variants — first turn and late turn), peer-message imperative as observation, Discord mention as recipient signal (two address variants), and multi-step constraint embedded in a chat ask. The expected verdict per fixture is stated against the pilot's `before-you-respond.md` classifier (default-PASS; SPEAK / ACK / ASK as the three non-silent exits).
- **FR-019**: Each fixture MUST carry a `source_shape` metadata field with value `"multica"` (TUR-8/9/10-derived) or `"discord"` (pilot-bot-session-derived). The runner MUST surface this field in both human-readable and machine-readable output and MUST allow filtering by it (e.g., `--source discord`), without changing pass/fail semantics for the unfiltered run.
- **FR-020**: The test suite MUST include a verdict-surface contract fixture asserting that the classifier adapter (FR-022) returns a structured object whose `verdict` field is one of the four typed values `{PASS, ACK, ASK, SPEAK}`, not a free-text sentinel string the model must spell (such as the `CC_CONNECT_SILENT_PASS` sentinel). A classifier output of `"PASS"` (bare string), `"__CC_CONNECT_SILENT_PASS___"`, `"__CC_CONNECT_SILENT_PASS____"`, or any other malformed sentinel variant MUST be rejected by the adapter's response-validation path and reported as a contract failure distinct from a verdict miscategorization. This fixture exists to gate against the pilot-bot-observed failure where bots emit the literal sentinel into Discord and defeat PASS suppression (Constitution Principle II). (Contract-test mechanism clarified 2026-05-25 — see Clarifications.)
- **FR-022**: The test suite MUST consume the classifier under test through a pluggable **classifier adapter** interface, with `turnaware admit --input <file>.json` (subprocess) as the bundled default adapter. The adapter takes a fixture-shaped JSON envelope as input and returns a structured result object containing at least: `verdict` (typed value per FR-020), `context_checked` (list of consulted context items), and `confidences` (per-verdict distribution). An adapter MUST also surface adapter-level errors (subprocess crash, malformed output, schema violation) as distinct from classifier-level mis-verdicts, so FR-020 and FR-001/002 failures are not confused. (Adapter boundary clarified 2026-05-25 — see Clarifications.)
- **FR-021**: The test suite MUST include at least one fixture per non-default Discord suppressor named in `before-you-respond.md`: `Self-caused` (the trigger is the receiving agent's own prior message), `Stale` (the operator has explicitly closed the session via `!stop` or equivalent since the trigger was sent), `Duplicate` (the receiving agent's own recent contributions already say what it would say), and `Covered` (a peer's latest message already provides the value-add the receiving agent would contribute). Each suppressor fixture pairs a trigger with the context items that activate the suppressor; the expected verdict is PASS, and the rationale names which suppressor is exercised.

### Key Entities

- **Fixture**: a self-contained test case. Holds a Multica-shaped envelope (trigger text + ordered context items as observed in TUR-8/9/10), the expected verdict (or set of acceptable verdicts), a source pointer (TUR-N issue + comment id, OR an explicit "predicted, not observed" marker), the failure mode or invariant exercised, and a one-line rationale for relevance.
- **Verdict**: one of PASS, ACK, ASK, SPEAK as defined by the TurnAware protocol. PASS is the only hard-stop verdict (no user-visible message permitted). The suite operates over this closed set.
- **Failure mode**: a named, observed-or-predicted defect (e.g., "substring trap", "trigger-only PASS", "context-trigger contradiction silently resolved", "constant confidence"). Each fixture is tagged with one. Output groups failures by mode.
- **Structural invariant**: a property the classifier must hold regardless of implementation (e.g., "PASS requires corroborating context"; "audit field reflects every consulted item"; "ASK is not the fallthrough verdict"). Each invariant is exercised by at least one fixture.
- **Test report**: the per-fixture record produced by a run. Holds the fields enumerated in FR-012 in both the machine-readable and human-readable forms (FR-013), and partitions failures by `a132ccc`-inherited failure modes vs. structural invariants (FR-016).
- **Fixture source provenance**: per fixture, either a runtime-observation pointer (the TUR-N issue and comment id where the verdict was observed) or a "predicted" flag indicating the case came from reading classifier code rather than from runtime evidence. For Discord-shape fixtures, the pointer is the pilot-bot session id plus message timestamp (e.g., `pilot-bot/6dc3f5aa#2026-05-13T00:32:22.818Z`) rather than a Multica comment id.
- **Conversation envelope shape**: an envelope is `multica`-shape (trigger text + ordered context items with comment-id source identifiers, as in TUR-8/9/10) or `discord`-shape (Discord message trigger + ordered last-N-messages context, each item carrying its author handle and timestamp, as in `~/github/pilot-bot/before-you-respond.md`). The classifier under test consumes both; the suite carries fixtures of both shapes and the runner does not branch on shape — only on the declared expected verdict and rationale metadata.
- **Discord suppressor**: one of the named PASS-producing rules in `~/github/pilot-bot/before-you-respond.md`: `Self-caused`, `Stale`, `Duplicate`, `Covered`. Each suppressor is exercised by at least one fixture (FR-021); fixtures name the suppressor in their rationale so a reader sees *which* rule fired, not just that PASS was correct.
- **Classifier adapter**: a pluggable bridge (FR-022) between the suite runner and a candidate classifier. Required interface: takes a fixture envelope (JSON object per FR-009) and returns a structured result `{verdict, context_checked, confidences, ...}` or an explicit adapter-level error. The bundled default adapter shells out to `turnaware admit --input <tmp>.json` and parses the JSON on stdout. A future in-process adapter would import the callable core and invoke it directly; both satisfy the same interface.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Running the test suite against commit `a132ccc` produces at least two failures: the false ACK case (FR-001) and the false PASS case (FR-002). Every representative per-verdict fixture (FR-003) passes against the same commit. No fixture flakes between runs.
- **SC-002**: Running the test suite against a candidate classifier that resolves both observed failure modes produces zero failures with no fixture edits. The same candidate's pass result is independently reproducible by a second reviewer on a separate machine with the same command.
- **SC-003**: The suite covers, at minimum, every required fixture class enumerated in FR-001 through FR-008. Each verdict (PASS, ACK, ASK, SPEAK) appears at least once as a "must keep passing" baseline, and each observed failure mode and each structural invariant is exercised by at least one fixture.
- **SC-004**: A contributor unfamiliar with the suite's internals can add a new fixture (envelope file + metadata) in under five minutes given an existing fixture as a template. No edits to the runner are required.
- **SC-005**: The full suite executes in under five seconds on a standard developer laptop. No network calls. No LLM calls. No external data fetched at run time.
- **SC-006**: For every failing fixture, the human-readable output names the fixture, the expected and observed verdicts, and the failure mode or invariant exercised — without requiring the reader to open the TUR-12 corpus comment or any TUR-8/9/10 thread.
- **SC-007**: The suite's pass/fail outcome can be consumed by an automated check (machine-readable form per FR-013) without parsing the human-readable form. Both forms agree on every fixture in every run.
- **SC-008**: The suite distinguishes runtime-observed failures from predicted-only failures in both forms of its output. A reviewer reading the report can count, without manual cross-reference, how many of the open failures are backed by runtime evidence.
- **SC-009**: The suite distinguishes Multica-shape fixtures from Discord-shape fixtures in both forms of output (per FR-019) and can be filtered to one source without changing the pass/fail semantics of the unfiltered run. A reviewer can answer "how many Discord-shape fixtures fail on the candidate" without manually grouping.
- **SC-010**: Every Discord-pilot-shape edge case enumerated in this spec is exercised by at least one fixture (FR-018). Each fixture's `source_shape="discord"` metadata is reflected in output. The four named Discord suppressors (`Self-caused`, `Stale`, `Duplicate`, `Covered` — FR-021) each have at least one PASS fixture, and the rationale field names which suppressor is being exercised.
- **SC-011**: The verdict-surface contract fixture (FR-020) fails on any classifier whose output for a PASS-shaped envelope is the literal string `"__CC_CONNECT_SILENT_PASS__"`, `"__CC_CONNECT_SILENT_PASS___"`, `"__CC_CONNECT_SILENT_PASS____"`, or any other malformed sentinel variant, and passes on any classifier returning the typed value `verdict="PASS"`. The fixture's failure message distinguishes "sentinel-leak surface violation" from "verdict miscategorization" so an implementer can tell which contract was broken.

This matches the public CLI's `--input <sample>.json` shape established in TUR-8 / TUR-9 / TUR-10.
- The fixture envelope shape mirrors the Multica-shaped envelopes used by the TUR-8 / TUR-9 / TUR-10 smoke runs (trigger text + ordered list of context items, each item carrying enough source identification to be cited in the audit field).
- The TUR-12 adversarial corpus comment (id `42611d71-af31-4881-9fd9-33161dac2ffb` on issue `b45da84e-023b-4f1a-bdb3-c7f2d03f30b9`) is the authoritative source for fixture content for Multica-shape cases. The pilot-bot session JSONL at `~/.claude/projects/-Users-zmll-github-pilot-bot/6dc3f5aa-6f96-4a55-a88e-9324f4467a5a.jsonl` (governed by `~/github/pilot-bot/before-you-respond.md`) is the authoritative source for Discord-shape cases. This spec does not re-investigate either pool; the implementer reconstructs envelopes from the trigger/context summaries documented in this spec and in those sources.
- For Discord-shape fixtures, the classifier under test is assumed to be the same TurnAware admission core, configured (or specialized) per the pilot's `before-you-respond.md` policy (default-PASS, named suppressors, peer-imperative-as-observation). The suite tests verdict outcomes against that policy, not against an arbitrary "what would Discord do." If a future TurnAware policy changes the default or the suppressor set, the Discord-shape fixtures and their expected verdicts MUST be updated alongside.
- The two observed failure modes (false ACK on `"comment back with results"`, false PASS on the fake-done trigger) were reproducible at commit `a132ccc` of `turnaware-0.1.0`. The suite encodes the *requirement* that those triggers produce the right verdict; if the classifier on `main` has since moved, the failure surface may have shifted, but the requirement does not.
- This spec is for the test suite only. It does not produce or modify the classifier itself. The implementation of classifier fixes is the responsibility of the classifier-completion lifecycle (TUR-11), which consumes this suite as its acceptance contract.
- The suite is provider- and model-agnostic. All observed failures are mechanism bugs in the deterministic keyword path; no fixture depends on LLM behaviour. A future LLM-based classifier is also tested against this suite as long as it exposes the same verdict surface.
- The repository's existing SpecKit workflow (`/speckit-specify` → `/speckit-clarify` → `/speckit-plan` → `/speckit-tasks` → `/speckit-implement`) is the production pathway. This file is the output of `/speckit-specify`; the runner and fixtures themselves are produced by `/speckit-implement` after `/speckit-plan` and `/speckit-tasks` have been completed.
- The spec directory name `003-classifier-test-suite` is assigned because `001-core-cli-mvp` and `002-admission-classifier` are already in flight on other branches; this spec is a standalone test-suite spec and does not co-own either of those lifecycles.
